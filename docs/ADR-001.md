# ADR-001 – Technology Stack and Architectural Direction

## Status

Accepted

---

## Context

The goal of this project is to build a production-grade personal website that:

- Demonstrates engineering standards
- Uses modern frontend architecture
- Is performant and accessible
- Supports multilingual content
- Supports light/dark theming
- Can be deployed with minimal operational overhead
- Remains scalable for future enhancements (e.g., Three.js integration)

The project should remain simple in scope but high in quality.

---

## Decision

The application will be built using:

- **Next.js (App Router)**
- **TypeScript (strict mode)**
- **Tailwind CSS (custom theme configuration)**
- Static-oriented deployment via Vercel

---

## Rationale

### Why Next.js?

- Industry-standard framework frequently required in frontend roles
- App Router provides modern architecture (layouts, server/client components)
- Built-in routing and metadata handling
- Strong performance optimizations out-of-the-box
- Easy CI/CD deployment
- Allows future expansion (server actions, API routes, ISR)

Even though SSR is not required initially, Next.js enables future evolution without architectural rewrites.

---

### Why Static-Oriented Deployment?

The website does not initially require dynamic SSR.

Static deployment:

- Reduces operational complexity
- Minimizes hosting costs
- Improves performance
- Simplifies maintenance

The architecture remains compatible with server-side features if required in the future.

---

### Why TypeScript (strict)?

- Improved reliability
- Clear contracts between components
- Better refactoring safety
- Production-level standards

---

### Why Tailwind CSS?

- Rapid UI development
- Design token customization
- Encourages consistency
- Enables building a small internal design system
- Supports dark mode out-of-the-box

The goal is to go beyond utility usage and create structured UI primitives.

---

## Consequences

### Positive

- Modern and production-ready stack
- Easy CI/CD integration
- High Lighthouse potential
- Scalable structure
- Demonstrates engineering maturity

### Negative / Trade-offs

- Slightly higher initial complexity than a simple static HTML site
- Requires understanding of App Router concepts
- Some framework abstraction overhead

These trade-offs are considered acceptable given the long-term benefits.

---

## Future Considerations

Potential future architectural extensions:

- Three.js-based interactive module
- MDX-powered blog
- Public design system documentation
- Analytics integration
- Performance experimentation

The current decision does not limit these extensions.
